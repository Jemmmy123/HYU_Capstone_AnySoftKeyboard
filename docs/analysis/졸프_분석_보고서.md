# AnySoftKeyboard 자동 추천 기능의 민감 정보 노출 경로 분석

## InputType 기반 보호 정책 분석 및 PoC 검증

작성일: 2026-05-19
담당: A, 분석 및 공격 담당

---

## 1. 개요

### 1.1 연구 배경

모바일 키보드 앱의 자동 추천 기능은 사용자가 입력한 단어, 이전 입력 이력, 연락처, 클립보드 등을 활용해 다음 입력을 빠르게 할 수 있도록 돕는다. 그러나 같은 기능은 ID, 비밀번호, PIN, 인증번호, 계좌번호, 카드번호 등 민감 정보가 학습되거나 추천 후보로 노출되는 경로가 될 수 있다.

특히 키보드 앱은 여러 앱의 입력 필드를 공통으로 처리하기 때문에, 한 앱에서 입력한 정보가 다른 앱의 입력 화면에서 추천될 수 있다. 따라서 자동 추천 기능은 단순한 사용성 기능이 아니라 입력 데이터 보호 정책과 직접 연결되는 보안 기능으로 분석할 필요가 있다.

### 1.2 연구 대상

분석 대상은 오픈소스 Android IME인 AnySoftKeyboard이다. 본 분석은 다음 경로의 Android IME 코드에 초점을 둔다.

- `ime/app/src/main/java/com/anysoftkeyboard/AnySoftKeyboard.java`
- `ime/app/src/main/java/com/anysoftkeyboard/ime/AnySoftKeyboardSuggestions.java`
- `ime/app/src/main/java/com/anysoftkeyboard/ime/AnySoftKeyboardIncognito.java`
- `ime/app/src/main/java/com/anysoftkeyboard/ime/AnySoftKeyboardClipboard.java`
- `ime/app/src/main/java/com/anysoftkeyboard/dictionaries/SuggestionsProvider.java`
- `ime/app/src/main/java/com/anysoftkeyboard/utils/IMEUtil.java`

### 1.3 연구 목표

1. AnySoftKeyboard가 `EditorInfo.inputType`과 `imeOptions`를 기반으로 추천 및 학습을 차단하는 방식을 분석한다.
2. 민감 입력 필드에서 추천 표시, 개인화 학습, 클립보드 추천이 노출될 수 있는 경로를 식별한다.
3. PoC 앱으로 원본 동작을 재현하고, 이후 B 담당자의 보안 패치 전후 효과를 정량적으로 평가한다.

## 2. 분석 방법론

본 분석은 코드 기반 정적 분석과 PoC 기반 동적 검증을 함께 수행하는 방식으로 설계하였다.

1. IME 진입점 추적
   - `AnySoftKeyboard.onStartInput()`과 `onStartInputView()`에서 시작해 상속 체인의 `super` 호출 흐름을 추적하였다.
   - 실제 입력 필드별 추천 차단 로직은 `onStartInput()`보다 `onStartInputView()`에 집중되어 있음을 확인하였다.

2. 핵심 상태 변수 식별
   - 추천 표시 여부: `mPredictionOn`
   - 자동 확정 여부: `mInputFieldSupportsAutoPick`, `mAutoCorrectOn`
   - 개인화 학습 차단 여부: `mIncognitoMode`

3. 입력 타입별 분기 분석
   - `TYPE_MASK_CLASS`, `TYPE_MASK_VARIATION`, `TYPE_MASK_FLAGS`를 기준으로 어떤 필드에서 추천이 꺼지는지 표로 정리하였다.

4. 학습 경로 추적
   - 입력 단어가 `checkAddToDictionaryWithAutoDictionary()`를 거쳐 `SuggestionsProvider.tryToLearnNewWord()` 및 `addWordToUserDictionary()`로 전달되는 경로를 확인하였다.
   - `mIncognitoMode`가 켜지면 자동 사전 및 사용자 사전 학습이 차단됨을 확인하였다.

5. 테스트 코드 확인
   - `IMEUtilTest`, `AnySoftKeyboardSuggestionsTest`, `AnySoftKeyboardClipboardTest`를 확인해 현재 동작이 의도된 설계인지, 테스트되지 않은 경계 조건인지 구분하였다.

## 3. 핵심 동작 분석

### 3.1 입력 필드 시작 시 호출 흐름

Android IME는 입력 필드가 시작될 때 `EditorInfo`를 키보드에 전달한다. AnySoftKeyboard에서는 이 정보를 이용해 키보드 레이아웃, 추천 표시 여부, 인코그니토 모드, 클립보드 추천 표시 등을 결정한다.

핵심 분석 대상은 다음 두 메서드이다.

```java
public void onStartInput(EditorInfo attribute, boolean restarting)
public void onStartInputView(EditorInfo attribute, boolean restarting)
```

`onStartInput()`은 주로 이전 입력 상태를 정리하는 역할을 한다. 반면 입력 필드의 `inputType`을 해석해 추천 표시 여부를 결정하는 핵심 로직은 `AnySoftKeyboardSuggestions.onStartInputView()`에 있다.

### 3.2 보호 메커니즘 A: 추천 표시 차단

`AnySoftKeyboardSuggestions.onStartInputView()`는 기본적으로 `mPredictionOn = true`로 시작한 뒤, 입력 타입에 따라 추천을 끈다.

```java
mPredictionOn = true;

switch (attribute.inputType & EditorInfo.TYPE_MASK_CLASS) {
  case EditorInfo.TYPE_CLASS_DATETIME:
  case EditorInfo.TYPE_CLASS_NUMBER:
  case EditorInfo.TYPE_CLASS_PHONE:
    mPredictionOn = false;
    break;

  case EditorInfo.TYPE_CLASS_TEXT:
    final int textVariation = attribute.inputType & EditorInfo.TYPE_MASK_VARIATION;
    switch (textVariation) {
      case EditorInfo.TYPE_TEXT_VARIATION_PASSWORD:
      case EditorInfo.TYPE_TEXT_VARIATION_VISIBLE_PASSWORD:
      case EditorInfo.TYPE_TEXT_VARIATION_WEB_PASSWORD:
        mPredictionOn = false;
        break;
    }
}
```

추가로 `TYPE_TEXT_FLAG_NO_SUGGESTIONS`가 설정된 경우 추천을 끄지만, 이 처리는 `IMEUtil.shouldHonorNoSuggestionsFlag()`의 결과에 의존한다.

```java
final int textFlag = attribute.inputType & EditorInfo.TYPE_MASK_FLAGS;
if (IMEUtil.shouldHonorNoSuggestionsFlag(textFlag)) {
  mPredictionOn = false;
}
```

### 3.3 보호 메커니즘 B: 개인화 학습 차단

`AnySoftKeyboardIncognito.onStartInputView()`는 특정 입력 필드에서 인코그니토 모드를 자동으로 켠다.

```java
if (isNoPersonalizedLearning(info) || isTextPassword(info) || isNumberPassword(info)) {
  setIncognito(true, false);
} else {
  setIncognito(mUserEnabledIncognito, false);
}
```

자동 인코그니토 진입 조건은 다음과 같다.

| 조건 | 의미 |
|---|---|
| `IME_FLAG_NO_PERSONALIZED_LEARNING` | 앱이 개인화 학습 금지를 요청 |
| `TYPE_TEXT_VARIATION_PASSWORD` | 텍스트 비밀번호 |
| `TYPE_TEXT_VARIATION_VISIBLE_PASSWORD` | 표시되는 텍스트 비밀번호 |
| `TYPE_TEXT_VARIATION_WEB_PASSWORD` | 웹 비밀번호 |
| `TYPE_CLASS_NUMBER | TYPE_NUMBER_VARIATION_PASSWORD` | 숫자 비밀번호, PIN |

인코그니토 모드가 켜지면 `SuggestionsProvider`에서 학습 경로가 차단된다.

```java
public boolean addWordToUserDictionary(String word) {
  if (mIncognitoMode) return false;
  ...
}

public boolean tryToLearnNewWord(CharSequence newWord, int frequencyDelta) {
  if (mIncognitoMode || !mNextWordEnabled) return false;
  ...
}
```

### 3.4 추천 차단과 학습 차단은 별도 정책이다

본 분석에서 중요한 점은 추천 차단과 학습 차단이 같은 기능이 아니라는 것이다.

| 보호 정책 | 담당 변수/모드 | 주요 효과 |
|---|---|---|
| 추천 표시 차단 | `mPredictionOn = false` | 후보 strip에 추천을 표시하지 않음 |
| 학습 차단 | `mIncognitoMode = true` | 사용자 사전, 자동 사전, 다음 단어 학습 차단 |

두 정책이 항상 함께 적용되는 것은 아니다. 예를 들어 텍스트 비밀번호는 추천과 학습이 모두 차단되지만, `IME_FLAG_NO_PERSONALIZED_LEARNING`은 학습만 차단하고 추천 표시는 계속될 수 있다.

### 5.2 F1: `NO_SUGGESTIONS` 조건부 무시

#### 내용

`TYPE_TEXT_FLAG_NO_SUGGESTIONS`는 앱이 키보드에게 추천을 표시하지 말라고 요청하는 플래그이다. 그러나 AnySoftKeyboard는 이 플래그가 `AUTO_CORRECT` 또는 `AUTO_COMPLETE`와 함께 설정되면 `NO_SUGGESTIONS`를 무시한다.

```java
if (hasAutoCorrect || hasAutoComplete) {
  return false;
}
```

이 동작은 단순 실수라기보다 의도된 정책으로 보인다. 코드 주석과 테스트는 Google Keep처럼 모순된 플래그를 보내는 앱에서 사용성을 유지하기 위해 `NO_SUGGESTIONS`를 무시한다고 설명한다.

#### 보안 의미

사용성 관점에서는 타당할 수 있지만, 보안 관점에서는 문제가 된다. 앱이 `NO_SUGGESTIONS`로 민감 입력 보호 의도를 표현했더라도, 다른 플래그 하나 때문에 키보드가 그 의도를 무시할 수 있기 때문이다.

#### PoC 시나리오

```java
editText.setInputType(
    InputType.TYPE_CLASS_TEXT
        | InputType.TYPE_TEXT_FLAG_NO_SUGGESTIONS
        | InputType.TYPE_TEXT_FLAG_AUTO_CORRECT);
```

관찰 항목:

- 추천 strip 표시 여부: 원본 AnySoftKeyboard에서 재현 완료
- 입력한 고유 문자열이 자동 추천 후보로 등장하는지 여부: 추가 검증 필요
- 같은 문자열을 다른 일반 텍스트 필드에서 다시 입력할 때 추천되는지 여부: 추가 검증 필요

#### 보고서 표현 기준

이 항목은 “구현 버그”라기보다 “보안 플래그와 사용성 플래그가 충돌할 때 사용성을 우선하는 정책 취약점”으로 표현하는 것이 정확하다.

#### PoC 결과

원본 AnySoftKeyboard에서 다음 결과를 확인하였다.

| PoC 필드 | 설정 | 관찰 결과 |
|---|---|---|
| T1 | 일반 텍스트 | 추천 strip 표시됨 |
| T4 | `TYPE_TEXT_FLAG_NO_SUGGESTIONS` | 추천 strip 표시 안 됨 |
| T5 | `TYPE_TEXT_FLAG_NO_SUGGESTIONS | TYPE_TEXT_FLAG_AUTO_CORRECT` | 추천 strip 표시됨 |
| T6 | `TYPE_TEXT_FLAG_NO_SUGGESTIONS | TYPE_TEXT_FLAG_AUTO_COMPLETE` | 추천 strip 표시됨 |

따라서 `NO_SUGGESTIONS`는 단독으로는 추천 차단에 사용되지만, `AUTO_CORRECT` 또는 `AUTO_COMPLETE`와 함께 설정될 경우 추천 strip 차단 효과가 사라지는 것을 확인하였다. 다만 이 케이스에서 입력 문자열이 실제로 개인 사전에 학습되어 다른 필드에서 재추천되는지는 아직 확인되지 않았으므로, 현재 F1의 검증 범위는 “추천 UI/policy 우회”로 제한한다.

### 5.3 F2: 숫자 비밀번호 필드의 클립보드 추천 마스킹 누락

#### 내용

`AnySoftKeyboardClipboard`는 최근 클립보드 내용을 keyboard strip에 표시한다. 텍스트 비밀번호 필드에서는 클립보드 내용을 `**********`로 마스킹하지만, 현재 마스킹 조건은 `isTextPassword()`만 사용한다.

```java
mSuggestionClipboardEntry.setClipboardText(
    mLastSyncedClipboardLabel,
    mLastSyncedClipboardEntryInSecureInput || isTextPassword(info));
```

반면 숫자 비밀번호 판단 함수인 `isNumberPassword()`는 `AnySoftKeyboardIncognito`와 키 미리보기 차단에는 사용되지만, 클립보드 마스킹 조건에는 포함되어 있지 않다.

#### 보안 의미

`TYPE_CLASS_NUMBER | TYPE_NUMBER_VARIATION_PASSWORD`로 설정된 PIN/숫자 비밀번호 필드에서 최근 클립보드 내용이 평문으로 표시될 가능성이 있다. 이는 추천 학습과 별개로, keyboard strip UI를 통한 직접 노출 경로이다.

#### PoC 시나리오

1. OS 클립보드에 `123456` 또는 `mySecretPin` 같은 값을 복사한다.
2. PoC 앱의 `TYPE_CLASS_NUMBER | TYPE_NUMBER_VARIATION_PASSWORD` 필드로 이동한다.
3. 키보드 상단 strip에 클립보드 값이 평문으로 표시되는지 확인한다.

예상 결과:

- 텍스트 비밀번호 필드: `**********` 표시
- 숫자 비밀번호 필드: 평문 표시 가능성

이 항목은 코드상 조건 누락이 분명하고, 데모 화면으로 보여주기 쉬워 PoC 우선순위가 높다.

#### PoC 결과

원본 AnySoftKeyboard에서 `Copy Clipboard Test` 버튼으로 `clipcasepin123456` 형식의 고유 값을 클립보드에 복사한 뒤 T1/T2/T3를 비교하였다. PoC 앱은 버튼을 누를 때마다 `clipcasepin123456`, `clipcasepin123457`처럼 값을 바꾸어 새 클립보드 이벤트를 발생시킨다. 단, 앱이 표시하는 Toast와 키보드의 클립보드 strip을 구분해야 한다.

재현 조건:

- AnySoftKeyboard의 OS clipboard sync 기능 활성화
- 일반 텍스트 필드 또는 비보안 컨텍스트에서 `Copy Clipboard Test` 버튼 클릭 후 즉시 T1/T2/T3 확인
- 화면 아래 Toast가 아니라 키보드 바로 위 클립보드 strip 기준으로 관찰

주의할 점은, 텍스트 비밀번호 필드가 포커스된 상태에서 클립보드 값이 복사되면 AnySoftKeyboard가 해당 클립보드 항목을 secure input에서 온 값으로 기록하여 다른 필드에서도 마스킹할 수 있다는 점이다. 본 F2 검증은 비보안 컨텍스트에서 복사된 클립보드 값이 숫자 비밀번호 필드에서 어떻게 표시되는지를 확인한다.

| PoC 필드 | 설정 | 관찰 결과 |
|---|---|---|
| T1 | 일반 텍스트 | 클립보드 strip에 `clipcasepin...` 평문 표시 |
| T2 | `TYPE_TEXT_VARIATION_PASSWORD` | 클립보드 strip에 `**********` 마스킹 표시 |
| T3 | `TYPE_CLASS_NUMBER | TYPE_NUMBER_VARIATION_PASSWORD` | 클립보드 strip에 `clipcasepin...` 평문 표시 |

따라서 텍스트 비밀번호 필드에서는 클립보드 값이 마스킹되지만, 숫자 비밀번호 필드에서는 동일한 클립보드 값이 평문으로 노출되는 것을 확인하였다.

### 5.4 F3: 웹/이메일/URI 입력 필드의 방어적 보호 부족

#### 내용

`TYPE_TEXT_VARIATION_WEB_EDIT_TEXT`, `TYPE_TEXT_VARIATION_EMAIL_ADDRESS`, `TYPE_TEXT_VARIATION_URI`는 비밀번호로 분류되지 않는다. 따라서 기본적으로 인코그니토 모드가 켜지지 않는다.

이 중 이메일과 URI는 자동 확정만 비활성화된다.

```java
case EditorInfo.TYPE_TEXT_VARIATION_EMAIL_ADDRESS:
case EditorInfo.TYPE_TEXT_VARIATION_URI:
case EditorInfo.TYPE_TEXT_VARIATION_WEB_EMAIL_ADDRESS:
  mInputFieldSupportsAutoPick = false;
  break;
```

그러나 `mPredictionOn` 자체는 꺼지지 않는다.

#### 보안 의미

이메일 주소는 로그인 ID로 사용되는 경우가 많고, URI 필드에는 초대 링크, API endpoint, 토큰이 포함된 URL 등이 입력될 수 있다. 웹뷰 기반 입력 필드가 정확한 password variation을 제공하지 않는 경우 `WEB_EDIT_TEXT`로 처리되어 일반 텍스트처럼 추천과 학습이 동작할 수 있다.

단, 이 항목은 “키보드가 Android 표준 password 필드를 놓쳤다”는 의미는 아니다. 더 정확한 표현은 다음과 같다.

> 앱이 민감한 웹/식별자 필드를 password variation으로 표시하지 않을 때, AnySoftKeyboard는 추가 휴리스틱 없이 일반 텍스트처럼 처리한다.

#### PoC 시나리오

- `TYPE_TEXT_VARIATION_WEB_EDIT_TEXT` 필드에 `leakcasewebtoken` 입력
- `TYPE_TEXT_VARIATION_EMAIL_ADDRESS` 필드에 고유 이메일 입력
- 일반 텍스트 필드로 이동 후 일부 prefix 입력
- 추천 후보 또는 학습 여부 확인

#### PoC 결과

원본 AnySoftKeyboard에서 `lea` prefix 입력 후 다음 결과를 확인하였다. 이 검증은 추천 strip 표시 여부를 확인한 것이며, 입력 문자열이 개인 사전에 학습되는지는 별도 검증이 필요하다.

| PoC 필드 | 설정 | 관찰 결과 |
|---|---|---|
| T7 | `TYPE_TEXT_VARIATION_WEB_EDIT_TEXT` | 추천 strip 표시됨 |
| T8 | `TYPE_TEXT_VARIATION_EMAIL_ADDRESS` | 추천 strip 표시됨 |

따라서 웹 텍스트 및 이메일 필드는 password variation이 아닌 경우 추천 표시가 유지된다. 이메일 필드의 경우 키보드 레이아웃은 이메일 입력에 맞게 바뀌지만, 후보 strip 자체는 유지되는 것을 확인하였다.

- [T7 WEB_EDIT_TEXT: 추천 strip 표시](screenshots/f3_t7_web_edit_text_suggestions.png)
- [T8 EMAIL_ADDRESS: 추천 strip 표시](screenshots/f3_t8_email_suggestions.png)

### 5.5 F4: `IME_FLAG_NO_PERSONALIZED_LEARNING`의 범위 제한

#### 내용

`IME_FLAG_NO_PERSONALIZED_LEARNING`은 인코그니토 모드를 켜서 학습을 차단한다. 그러나 `mPredictionOn`을 끄는 조건에는 포함되어 있지 않다.

#### 보안 의미

앱이 개인화 학습 금지를 요청해도 이미 다른 곳에서 학습된 추천 후보는 계속 표시될 수 있다. 플래그 이름 그대로 해석하면 이는 Android 정책 위반은 아닐 수 있다. 이 플래그는 “학습하지 말라”는 의미이지 “추천을 모두 숨겨라”는 의미는 아니기 때문이다.

그러나 민감 입력 UX 관점에서는 혼동이 생긴다. 사용자는 보안 입력 필드라고 기대하는 화면에서 개인 추천 후보가 노출되는 것을 볼 수 있다.

#### PoC 시나리오

1. 일반 텍스트 필드에서 고유 문자열을 학습시킨다.
2. `IME_FLAG_NO_PERSONALIZED_LEARNING`만 설정된 필드로 이동한다.
3. prefix 입력 시 기존 추천 후보가 표시되는지 확인한다.

#### PoC 결과

T9 `IME_FLAG_NO_PERSONALIZED_LEARNING` 필드에서 `lea` prefix 입력 시 추천 strip이 표시되는 것을 확인하였다. 이는 해당 플래그가 개인화 학습 차단에는 사용되지만, 추천 strip 표시 자체를 차단하지는 않는다는 코드 분석과 일치한다.

증거 스크린샷:

- [T9 NO_PERSONALIZED_LEARNING: 추천 strip 표시](screenshots/f4_t9_no_personalized_learning_suggestions.png)

### 5.6 F5: `hintText`/label 기반 민감 필드 휴리스틱 부재

#### 내용

현재 보호 로직은 주로 `inputType`과 `imeOptions`에 의존한다. `EditorInfo.hintText` 또는 앱의 label에 포함된 의미 정보는 보호 판단에 사용되지 않는다.

#### 보안 의미

실제 앱에서는 개발자가 `inputType`을 부정확하게 설정하고, 화면 hint만 “비밀번호”, “PIN”, “카드번호”, “인증번호”처럼 표시하는 경우가 있다. 이 경우 키보드는 해당 필드를 일반 텍스트로 처리한다.

#### PoC 시나리오

```java
editText.setInputType(InputType.TYPE_CLASS_TEXT);
editText.setHint("비밀번호");
```

예상 관찰 항목:

- 추천 strip 표시 여부
- 입력 문자열 학습 여부
- B 패치 후 hint 기반 차단 여부

#### PoC 결과

T10은 `inputType`이 일반 텍스트이고 hint만 `비밀번호`로 설정된 필드이다. 원본 AnySoftKeyboard에서 `lea` prefix 입력 시 추천 strip이 표시되는 것을 확인하였다.

따라서 현재 보호 정책은 `hintText`의 의미를 민감 필드 판단에 사용하지 않는다. 다만 이 결과는 Android 표준 password inputType을 사용하지 않은 앱 구현에 대한 방어적 보강 필요성을 의미하며, 표준 비밀번호 필드 처리 실패를 의미하지는 않는다.

증거 스크린샷:

- [T10 hint=비밀번호: 추천 strip 표시](screenshots/f5_t10_hint_password_suggestions.png)

### 5.7 F6: `TYPE_NULL` 또는 잘못 설정된 `inputType` 신뢰 문제

#### 내용

`inputType` class가 명확하지 않은 경우 AnySoftKeyboard는 기본 텍스트 모드로 진입한다.

```java
default:
  mAutoSpace = mPrefsAutoSpace;
  getKeyboardSwitcher().setKeyboardMode(KeyboardSwitcher.INPUT_MODE_TEXT, attribute, restarting);
```

#### 보안 의미

일부 커스텀 입력 필드, 터미널형 앱, 잘못 구현된 앱은 `TYPE_NULL` 또는 불완전한 `inputType`을 전달할 수 있다. 이 경우 키보드는 앱이 제공한 타입을 신뢰하며 추가 방어를 수행하지 않는다.

#### PoC 결과

T11 `TYPE_NULL` 필드에서 `lea` prefix 입력 시 키보드는 표시되었지만, T6~T10과 같은 추천 후보 strip은 표시되지 않았다. 따라서 현재 PoC 기준으로는 `TYPE_NULL`에서 추천 표시 취약점은 재현되지 않았다.

이 항목은 취약점이라기보다 방어적 설계 검토 항목으로 유지한다. B 패치 범위에서는 우선순위를 낮추고, 커스텀 입력 필드에서 민감 정보를 다루는 실제 앱 사례가 있을 때 추가 분석하는 것이 적절하다.

증거 스크린샷:

- [T11 TYPE_NULL: 추천 후보 strip 미표시](screenshots/f6_t11_type_null_no_suggestions.png)

이 항목은 단독 취약점이라기보다 방어적 설계 부족에 가깝다.

## 6. 이전 분석 대비 정정 사항

초기 분석에서 다음 표현은 수정이 필요하다.

### 6.1 일반 숫자 필드가 자동 학습된다는 표현은 보류

`TYPE_CLASS_NUMBER`와 `TYPE_CLASS_PHONE`은 `mPredictionOn = false`가 된다. 자동 학습 경로는 일반적으로 `isCurrentlyPredicting()` 상태를 전제로 하기 때문에, 단순히 인코그니토 모드가 켜지지 않는다는 이유만으로 “숫자 정보가 학습된다”고 단정하면 안 된다.

따라서 카드번호/계좌번호 같은 일반 숫자 필드는 다음처럼 표현한다.

> 추천 표시는 차단되지만 인코그니토 정책과는 분리되어 있다. 실제 학습 여부는 PoC로 별도 검증한다.

### 6.2 이메일 전체가 항상 학습된다는 표현은 완화

이메일과 URI는 `@`, `.`, `/`, `:` 같은 구분 문자를 포함한다. WordComposer와 사전 학습 로직이 전체 문자열을 하나의 단어로 학습하는지, 일부 token만 학습하는지는 동적 검증이 필요하다.

보고서에서는 “전체 이메일이 반드시 학습된다”보다 “식별자 또는 일부 token이 추천/학습될 가능성이 있다”로 표현한다.

### 6.3 `NO_SUGGESTIONS | AUTO_CORRECT`는 버그보다 정책 충돌

해당 동작은 코드와 테스트에 명시되어 있으므로 “개발자가 실수로 놓친 버그”라고 표현하기 어렵다. 대신 다음 표현이 적절하다.

> 보안 플래그와 사용성 플래그가 충돌할 때 사용성을 우선하여 보안 의도가 약화되는 정책 취약점.

## 7. PoC 앱 설계

### 7.1 테스트 필드 목록

PoC 앱은 상단에 재현 결과 요약을 표시하고, 아래에 여러 `EditText`를 배치하여 각 필드의 `inputType`과 `imeOptions`를 다르게 설정한다. 발표 흐름을 줄이기 위해 T1~T10을 메인 테스트 필드로 두고, 재현 우선순위가 낮은 T11은 optional edge case로 접어 둔다.

| ID | 필드 설정 | 목적 | 예상 결과 |
|---|---|---|---|
| T1 | 일반 텍스트 | 비교군 | 추천 활성 |
| T2 | `TYPE_TEXT_VARIATION_PASSWORD` | 텍스트 비밀번호 baseline | 추천/학습 차단, 클립보드 마스킹 확인 |
| T3 | `TYPE_CLASS_NUMBER | TYPE_NUMBER_VARIATION_PASSWORD` | 숫자 비밀번호 baseline | 추천/학습 차단, 클립보드 마스킹 확인 |
| T4 | `TYPE_TEXT_FLAG_NO_SUGGESTIONS` | 정상 플래그 baseline | 추천 차단 |
| T5 | `NO_SUGGESTIONS | AUTO_CORRECT` | F1 검증 | 추천 활성 여부 확인 |
| T6 | `NO_SUGGESTIONS | AUTO_COMPLETE` | F1 확장 검증 | 추천 활성 여부 확인 |
| T7 | `TYPE_TEXT_VARIATION_WEB_EDIT_TEXT` | F3 검증 | 추천/학습 여부 확인 |
| T8 | `TYPE_TEXT_VARIATION_EMAIL_ADDRESS` | F3 검증 | ID/token 추천 여부 확인 |
| T9 | `IME_FLAG_NO_PERSONALIZED_LEARNING` | F4 검증 | 학습 차단, 기존 추천 표시 여부 확인 |
| T10 | 일반 text + hint=`비밀번호` | F5 검증 | 휴리스틱 부재 확인 |
| T11 | `TYPE_NULL` | F6 검증 | optional edge case, 후보 추천 strip 미재현 |

### 7.2 실험 절차

1. AnySoftKeyboard 원본 버전을 설치하고 기본 키보드로 설정한다.
2. PoC 앱을 실행한다.
3. 각 테스트 필드에 고유 문자열을 입력한다.
   - 예: `leakcasealphaone`, `leakcasewebtoken`, `leakcaseemail@example.com`
4. 일반 텍스트 비교군인 T1 필드로 이동한다.
5. 고유 문자열의 prefix를 입력해 추천 후보 표시 여부를 기록한다.
6. 클립보드 테스트는 별도로 수행한다.
   - OS 클립보드에 고유 문자열 복사
   - 텍스트 비밀번호 필드와 숫자 비밀번호 필드에서 strip 표시 비교
7. 동일한 절차를 B 담당자의 패치 버전에서 반복한다.

### 7.3 기록 항목

각 테스트 케이스마다 다음 항목을 표로 기록한다.

| 항목 | 설명 |
|---|---|
| inputType / imeOptions | PoC 필드 설정 |
| 추천 strip 표시 | 후보 UI가 보이는지 |
| 입력 중 추천 후보 | 현재 필드에서 추천이 나오는지 |
| 재입력 추천 | 다른 필드에서 prefix 입력 시 다시 추천되는지 |
| 클립보드 strip 노출 | 클립보드 값이 평문/마스킹 중 무엇으로 보이는지 |
| 원본 결과 | 패치 전 결과 |
| 패치 결과 | 패치 후 결과 |
| 차단 성공 여부 | 개선 여부 |

## 8. 보안 강화 제안

### 8.1 보안 우선 모드에서 `NO_SUGGESTIONS` 무조건 존중

현재 `NO_SUGGESTIONS`는 `AUTO_CORRECT` 또는 `AUTO_COMPLETE`와 충돌하면 무시된다. 보안 강화 버전에서는 다음 중 하나를 선택할 수 있다.

1. 항상 `NO_SUGGESTIONS`를 우선한다.
2. 설정 옵션으로 “보안 우선 추천 차단 모드”를 제공한다.
3. `NO_SUGGESTIONS`가 설정된 경우 추천은 유지하더라도 학습은 무조건 차단한다.

졸업 프로젝트의 목표를 고려하면 3번이 사용성 손실을 줄이면서 보안 개선 효과를 보이기 좋다.

### 8.2 클립보드 마스킹 조건에 숫자 비밀번호 포함

`AnySoftKeyboardClipboard`의 마스킹 조건에 `isNumberPassword(info)`를 포함한다.

개선 방향:

```java
isTextPassword(info) || isNumberPassword(info)
```

이 변경은 범위가 작고 PoC로 효과를 명확히 입증할 수 있다.

### 8.3 민감 필드 감지 공통 함수 도입

현재 텍스트 비밀번호, 숫자 비밀번호, 개인화 학습 금지, 클립보드 마스킹 판단이 여러 클래스에 흩어져 있다. 공통 함수로 묶으면 정책 불일치를 줄일 수 있다.

예시:

```java
isSensitiveInput(EditorInfo info)
shouldDisablePrediction(EditorInfo info)
shouldDisablePersonalizedLearning(EditorInfo info)
shouldMaskClipboardSuggestion(EditorInfo info)
```

### 8.4 `hintText` 기반 휴리스틱 추가

다음 키워드가 `EditorInfo.hintText`에 포함되면 추천 또는 학습을 차단한다.

| 언어 | 키워드 예시 |
|---|---|
| 한국어 | 비밀번호, 암호, 패스워드, 핀, PIN, 카드번호, 계좌번호, 인증번호 |
| 영어 | password, passcode, pin, otp, card number, account number, verification code |

휴리스틱은 오탐 가능성이 있으므로 추천 표시를 완전히 끄기보다 학습 차단부터 적용하는 방식이 안전하다.

### 8.5 민감 패턴 학습 방지

입력된 단어가 다음 패턴을 만족하면 자동 사전 학습을 거부한다.

- 긴 영숫자 혼합 문자열
- 6자리 OTP 후보
- 13~19자리 카드번호 후보
- URL token 또는 query parameter 형태
- 이메일 주소 형태

이 기능은 inputType이 잘못 설정된 앱에서도 일부 보호를 제공한다.

## 9. 평가 지표

### 9.1 정량 평가

| 지표 | 계산 방식 |

| 추천 차단률 | 차단된 추천 케이스 수 / 전체 민감 케이스 수 |

- 앱이 잘못된 inputType을 제공해도 방어적 보호가 작동하는가?
- 일반 텍스트 입력의 사용성을 과도하게 해치지 않는가?
- 사용자에게 민감 입력 보호 상태가 명확히 전달되는가?

## 10. 결론

AnySoftKeyboard는 비밀번호 필드와 숫자 비밀번호 필드에 대해 추천 차단과 학습 차단을 모두 적용하고 있으며, 기본적인 민감 입력 보호 구조를 갖추고 있다. 그러나 보호 정책은 입력 타입과 플래그 조합에 따라 일관되지 않게 적용된다.

가장 중요한 발견은 `TYPE_TEXT_FLAG_NO_SUGGESTIONS`가 `AUTO_CORRECT` 또는 `AUTO_COMPLETE`와 함께 설정될 때 무시된다는 점이다. 이는 기존 테스트에 의해 의도된 동작으로 관리되고 있지만, 보안 관점에서는 앱의 추천 차단 의도가 약화되는 정책 취약점이 된다.

또한 클립보드 추천 마스킹 로직이 텍스트 비밀번호만 고려하고 숫자 비밀번호를 고려하지 않는 점은 직접적인 UI 노출 경로로 볼 수 있다. 이 항목은 PoC 재현과 패치 효과 입증이 비교적 명확하므로 우선 검증 대상으로 삼는다.

추가 PoC에서는 웹 텍스트, 이메일, 개인화 학습 금지, hint 기반 비밀번호 필드에서도 추천 strip이 표시되는 것을 확인하였다. 반면 `TYPE_NULL` 필드는 키보드는 표시되지만 후보 추천 strip은 재현되지 않아 방어 설계 검토 항목으로 낮은 우선순위에 둔다.

본 프로젝트는 위 발견 사항을 PoC 앱으로 재현하고, B 담당자의 보안 강화 패치 이후 추천 차단률, 학습 차단률, 클립보드 마스킹률을 비교해 자동 추천 기능의 보안성 개선 효과를 정량적으로 평가한다.
