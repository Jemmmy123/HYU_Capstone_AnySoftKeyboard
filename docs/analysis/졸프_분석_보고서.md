# 졸업 프로젝트 분석 보고서

## AnySoftKeyboard 자동 추천 기능의 보안 취약점 분석

---

## 1. 개요

### 1.1 연구 배경

키보드 앱의 자동 추천 기능은 사용 편의성을 위해 제공되지만, 입력된 정보를 학습하고 추천 후보로 노출하는 과정에서 ID, 패스워드 등 민감 정보가 유출될 수 있는 경로로 활용될 수 있다.

### 1.2 연구 대상

오픈소스 안드로이드 키보드 앱 **AnySoftKeyboard** (Apache 2.0 라이선스). 약 15년간 유지보수된 안정적인 프로젝트로, 자동 추천 기능의 전형적인 구현을 포함하고 있다.

### 1.3 연구 목표

1. AnySoftKeyboard의 자동 추천 동작을 분석하여 민감 정보 유출 가능 경로를 식별한다.
2. 식별된 취약점을 보완하는 보안 강화 기능을 구현한다.
3. 강화 전후의 효과를 정량적으로 비교 평가한다.

---

## 2. 분석 방법론

본 연구는 다음 절차로 코드를 분석하였다.

1. **상속 체인 추적**: AnySoftKeyboard 클래스에서 시작하여 super 호출을 따라 상위 클래스의 콜백 메서드를 단계적으로 분석
2. **핵심 변수 식별**: 추천 기능의 ON/OFF를 결정하는 변수를 발견하고 전역 검색으로 모든 참조 지점 추적
3. **호출 관계 분석**: Find Usages 기능을 활용해 보호 로직의 호출 위치를 매핑
4. **분기 로직 분석**: 각 inputType에 대한 키보드의 처리 방식을 카탈로그화
5. **표준 비교**: 안드로이드 공식 문서의 보안 플래그 명세와 실제 구현을 비교
6. **테스트 코드 분석**: 메인테이너의 설계 의도를 unit test 케이스에서 역추적
7. **격차 분석**: 두 보호 메커니즘 간의 적용 범위를 비교하여 보호 격차 도출

## 3. 분석 결과

### 3.1 두 가지 보호 메커니즘 발견

AnySoftKeyboard는 자동 추천의 보안 위협을 인지하고 두 가지 독립적인 보호 메커니즘을 구현하고 있다.

#### 메커니즘 A: 추천 표시 차단 (mPredictionOn)

- **위치**: AnySoftKeyboardSuggestions.onStartInputView()
- **목적**: 추천 후보를 사용자에게 표시하지 않음
- **활성화 조건**: inputType이 PASSWORD 3종, NUMBER, PHONE, DATETIME 중 하나거나, NO_SUGGESTIONS 플래그가 설정됨 (단, 조건부 우회 존재)

#### 메커니즘 B: 학습 차단 (Incognito Mode)

- **위치**: AnySoftKeyboardIncognito.onStartInputView()
- **목적**: 입력된 단어를 사전에 학습하지 않으며, 화면에 워터마크를 표시
- **활성화 조건 (OR)**: NO_PERSONALIZED_LEARNING 플래그, 텍스트 비밀번호, 숫자 비밀번호, 또는 사용자 수동 설정

### 3.2 핵심 코드

#### 메커니즘 A의 핵심 분기 로직

`AnySoftKeyboardSuggestions.onStartInputView()` 내부:

```java
mPredictionOn = true;  // 기본값

switch (inputType & TYPE_MASK_CLASS) {
    case TYPE_CLASS_NUMBER:    mPredictionOn = false; break;
    case TYPE_CLASS_PHONE:     mPredictionOn = false; break;
    case TYPE_CLASS_DATETIME:  mPredictionOn = false; break;
    case TYPE_CLASS_TEXT:
        switch (textVariation) {
            case TYPE_TEXT_VARIATION_PASSWORD:
            case TYPE_TEXT_VARIATION_VISIBLE_PASSWORD:
            case TYPE_TEXT_VARIATION_WEB_PASSWORD:
                mPredictionOn = false;
                break;
            case TYPE_TEXT_VARIATION_EMAIL_ADDRESS:
            case TYPE_TEXT_VARIATION_URI:
            case TYPE_TEXT_VARIATION_WEB_EMAIL_ADDRESS:
                // 자동 확정만 끔. mPredictionOn은 그대로 true
                mInputFieldSupportsAutoPick = false;
                break;
        }
}

// NO_SUGGESTIONS 플래그 처리
if (IMEUtil.shouldHonorNoSuggestionsFlag(textFlag)) {
    mPredictionOn = false;
}
```

#### NO_SUGGESTIONS 조건부 우회 로직

`IMEUtil.shouldHonorNoSuggestionsFlag()`:

```java
if (hasAutoCorrect || hasAutoComplete) {
    Logger.d(TAG, "Ignoring TYPE_TEXT_FLAG_NO_SUGGESTIONS due to contradictory flags...");
    return false;
}
```

#### 메커니즘 B의 활성화 조건

`AnySoftKeyboardIncognito.onStartInputView()`:

```java
if (isNoPersonalizedLearning(info) || isTextPassword(info) || isNumberPassword(info)) {
    setIncognito(true, false);
} else {
    setIncognito(mUserEnabledIncognito, false);
}
```

#### 비밀번호 인식 범위

`isTextPassword()`:

```java
return switch (textVariation) {
    case TYPE_TEXT_VARIATION_PASSWORD,
         TYPE_TEXT_VARIATION_WEB_PASSWORD,
         TYPE_TEXT_VARIATION_VISIBLE_PASSWORD -> true;
    default -> false;
};
```

`isNumberPassword()`:

```java
private static final int NUMBER_INCOGNITO_TYPE =
    EditorInfo.TYPE_CLASS_NUMBER | EditorInfo.TYPE_NUMBER_VARIATION_PASSWORD;

protected static boolean isNumberPassword(EditorInfo info) {
    return (info.inputType & NUMBER_INCOGNITO_TYPE) == NUMBER_INCOGNITO_TYPE;
}
```

#### Incognito 모드 적용 범위

`setIncognito()`:

```java
protected void setIncognito(boolean enable, boolean byUser) {
    getSuggest().setIncognitoMode(enable);              // 추천 엔진
    getQuickKeyHistoryRecords().setIncognitoMode(...);  // 입력 이력 기록
    setupInputViewWatermark();                          // 화면 워터마크
}
```

## 4. 발견된 취약점

본 연구는 총 6개의 보안 취약점을 식별하였다.

### 4.1 취약점 #1: 이메일/URI 필드 완전 무방비

**격차**: 두 메커니즘 모두 적용되지 않는 경우가 존재한다.

`AnySoftKeyboardSuggestions`에서 이메일과 URI 필드는 mPredictionOn을 변경하지 않고 자동 확정 기능만 비활성화한다. 동시에 `AnySoftKeyboardIncognito.isTextPassword()`는 이메일을 비밀번호로 인식하지 않으므로 Incognito 모드도 활성화되지 않는다. 코드 내 로그 메시지가 이를 명시한다.

```
"An internet input with has prediction but no auto-pick"
```

**공격 시나리오**: 이메일 주소는 로그인 ID로 흔히 사용된다. 사용자가 이메일을 입력하면 사전에 학습되며, 이후 일반 입력란에서 한두 글자만 입력해도 전체 이메일이 추천 후보로 노출되어 주변인에게 ID가 유출될 수 있다.

### 4.2 취약점 #2: NO_SUGGESTIONS 조건부 우회 (핵심 발견)

**격차**: AUTO_CORRECT 또는 AUTO_COMPLETE 플래그가 함께 설정된 경우, NO_SUGGESTIONS가 의도적으로 무시된다.

플래그 조합별 동작은 다음과 같다.

- NO_SUGGESTIONS 단독: 정상적으로 추천 차단
- NO_SUGGESTIONS + AUTO_CORRECT: 추천 활성화 (우회)
- NO_SUGGESTIONS + AUTO_COMPLETE: 추천 활성화 (우회)
- NO_SUGGESTIONS + 두 플래그 모두: 추천 활성화 (우회)

이는 단순 버그가 아니며, 다음 두 가지 증거로 의도된 설계임을 확인할 수 있다.

1. 코드 내 로그 메시지: `"Ignoring TYPE_TEXT_FLAG_NO_SUGGESTIONS due to contradictory flags"`
2. Unit test 케이스명: `testShouldHonorNoSuggestionsFlag_GoogleKeepScenario`

메인테이너는 Google Keep 등 정상 앱의 사용성을 보장하기 위해 이러한 우회 로직을 도입한 것으로 보인다. 그러나 보안 관점에서는 안드로이드 표준 플래그를 키보드가 일방적으로 무력화하는 우회 경로(bypass path)를 제공한다.

**공격 시나리오**: 악의적 앱이 비밀번호 필드를 다음과 같이 설정할 수 있다.

```java
editText.setInputType(
    InputType.TYPE_CLASS_TEXT
    | InputType.TYPE_TEXT_FLAG_NO_SUGGESTIONS  // 보안 의도 표시
    | InputType.TYPE_TEXT_FLAG_AUTO_CORRECT    // 무력화 트리거
);
```

키보드는 NO_SUGGESTIONS를 무시하고 추천/학습을 활성화하며, 사용자가 입력한 민감 정보가 학습된다.

### 4.3 취약점 #3: 일반 NUMBER/PHONE 학습 차단 부재

**격차**: 메커니즘 A는 적용되지만 메커니즘 B는 적용되지 않는다.

`isNumberPassword()`는 `TYPE_CLASS_NUMBER`와 `TYPE_NUMBER_VARIATION_PASSWORD`가 모두 설정된 경우(명시적 PIN 입력)만 인식한다. 그러나 현실의 민감 숫자 정보는 사용자가 입력값을 확인해야 하므로 PASSWORD variation을 명시하지 않는 경우가 대부분이다.

**보호되지 않는 케이스**:

- 신용카드 번호 (TYPE_CLASS_NUMBER만 설정)
- 계좌번호
- 주민등록번호
- 일반 전화번호 (TYPE_CLASS_PHONE)
- 인증번호(OTP)

**공격 시나리오**: 모바일 결제 앱이 카드번호 입력 필드를 `TYPE_CLASS_NUMBER`로만 설정한 경우, 추천 후보 자체는 표시되지 않으나(메커니즘 A) 사전에는 학습된다(메커니즘 B 미적용). 이후 다른 입력란에서 첫 자리 숫자만 입력해도 카드번호가 추천으로 노출될 수 있다.

### 4.4 취약점 #4: NO_PERSONALIZED_LEARNING의 비대칭

**격차**: 학습은 차단되지만 추천 표시는 계속된다.

`NO_PERSONALIZED_LEARNING` 플래그는 메커니즘 B(Incognito)만 활성화하고 메커니즘 A(mPredictionOn)에는 영향을 주지 않는다. 결과적으로 학습은 차단되지만, 다른 곳에서 학습된 추천 결과는 그대로 노출된다.

**의미적 모순**: 앱이 "개인화 학습 안 함" 의도를 표시했음에도, 화면에는 다른 사용자 학습 결과가 추천으로 표시되어 앱의 보안 의도와 실제 동작이 불일치한다.

### 4.5 취약점 #5: hint/label 휴리스틱 부재

**격차**: 두 메커니즘 모두 inputType만 검사하며, hint, label 등 보조 정보는 활용하지 않는다.

**공격 시나리오**: 잘못 구현된 앱이 일반 텍스트 inputType에 hint로만 "비밀번호를 입력하세요"라고 표시한 경우, AnySoftKeyboard는 이를 일반 텍스트로 인식하여 학습 및 추천을 활성화한다.

### 4.6 취약점 #6: inputType 신뢰 문제

**격차**: 키보드는 앱이 제공한 inputType을 무조건 신뢰하며 추가 검증을 수행하지 않는다.

악의적 또는 실수로 inputType이 잘못 설정된 경우 키보드 측에서 이를 탐지할 수 있는 방어 로직이 부재한다.

## 5. 핵심 학술 발견: 보호 정책의 비일관성

본 연구의 핵심 학술적 발견은, AnySoftKeyboard의 두 보호 메커니즘이 일관되지 않게 적용되어 다음 네 가지 보호 격차(Protection Gap)가 존재한다는 점이다.

### 5.1 메커니즘 비대칭

같은 보호 의도를 가진 inputType이라도 두 메커니즘 중 한쪽만 적용되는 경우가 다수 존재한다.

- 비밀번호 3종: 메커니즘 A, B 모두 적용 (안전)
- NUMBER, PHONE: 메커니즘 A만 적용 (학습 노출)
- EMAIL, URI: 두 메커니즘 모두 미적용 (완전 무방비)

### 5.2 플래그 처리 정책 비일관

안드로이드 표준 보안 플래그가 일관되지 않게 처리된다.

- NO_SUGGESTIONS: 조건부 무시 (사용성 우선)
- NO_PERSONALIZED_LEARNING: 무조건 존중 (보안 우선)

두 플래그 모두 사용자 데이터 보호라는 동일한 의도를 가지나, 처리 정책은 상반된다.

### 5.3 의미 격차

추천 차단과 학습 차단이 별개 메커니즘으로 분리되어 있어, 한쪽만 적용되는 비대칭 상태가 다수 발생한다. 이는 앱 개발자가 의도한 보안 정책과 키보드의 실제 동작 사이에 의미적 불일치를 야기한다.

### 5.4 검증 격차

오직 inputType이라는 단일 신호에만 의존하며, hint, label 등 보조 정보나 휴리스틱 기반 탐지 로직이 부재하다.

## 6. 보안 강화 제안

본 연구는 위 격차를 통합적으로 해결하는 보안 강화 패치를 다음과 같이 제안한다.

### 6.1 NO_SUGGESTIONS 우회 차단

**대상**: `IMEUtil.shouldHonorNoSuggestionsFlag()`

AUTO_CORRECT/AUTO_COMPLETE 검사 로직을 제거하거나, 사용자 옵션으로 보안 모드를 제공한다. 보안 모드에서는 NO_SUGGESTIONS를 무조건 존중한다.

### 6.2 이메일/URI 학습 차단

**대상**: `AnySoftKeyboardIncognito.isTextPassword()` 확장 또는 새 검사 메서드 추가

이메일, URI 입력 시 Incognito 모드를 자동 진입하도록 확장한다. 사용자 옵션으로 켜고 끌 수 있도록 한다.

### 6.3 일반 NUMBER 학습 차단

**대상**: `AnySoftKeyboardIncognito.isNumberPassword()` 확장

`TYPE_CLASS_NUMBER`이면서 일정 길이 이상(예: 신용카드 16자리)인 경우 자동 Incognito를 활성화한다.

### 6.4 hint/label 휴리스틱 검사

**대상**: 두 메커니즘 진입점에 휴리스틱 검사 추가

`attribute.hintText`에 "비밀번호", "password", "PIN", "카드번호" 등 민감 키워드가 포함된 경우 보호 활성화.

### 6.5 사용자 알림 UI

민감 필드로 자동 인식되어 학습/추천이 차단된 경우, 사용자에게 명확한 시각적 피드백(워터마크, 토스트 등)을 제공한다.

## 7. 평가 계획

본 연구의 효과는 다음 절차로 평가한다.

### 7.1 PoC 안드로이드 앱 제작

다양한 inputType과 플래그 조합을 가진 EditText 8종을 포함한 테스트 앱을 제작한다.

| 케이스 | 설정 | 졸프 입증 대상 |
|---|---|---|
| 1 | TYPE_TEXT_VARIATION_PASSWORD | Baseline (정상 동작) |
| 2 | TYPE_TEXT_VARIATION_EMAIL_ADDRESS | 취약점 #1 |
| 3 | TYPE_CLASS_NUMBER (신용카드 시뮬) | 취약점 #3 |
| 4 | NO_SUGGESTIONS 단독 | 정상 동작 일부 |
| 5 | NO_SUGGESTIONS + AUTO_CORRECT | 취약점 #2 (핵심) |
| 6 | NO_PERSONALIZED_LEARNING 단독 | 취약점 #4 |
| 7 | 일반 text + hint="비밀번호" | 취약점 #5 |
| 8 | 일반 text (baseline) | 비교군 |

### 7.2 실험 절차

1. 각 EditText에 고유 식별 문자열 입력 (예: LEAKCASE2EMAIL@test.com)
2. 일반 입력란으로 이동하여 일부 글자 입력
3. 추천 후보 목록에 식별 문자열이 노출되는지 관찰
4. 결과를 스크린샷 및 logcat으로 기록

### 7.3 강화 전후 비교

원본 AnySoftKeyboard와 패치된 버전에서 동일한 PoC 실험을 수행하여, 차단된 취약점 수를 정량적으로 비교한다.

## 8. 결론

AnySoftKeyboard는 자동 추천 기능의 보안 위협을 인지하고 두 가지 보호 메커니즘(추천 차단, 학습 차단)을 도입하였다. 그러나 두 메커니즘은 일관되게 적용되지 않으며, 본 연구는 6개의 구체적 취약점과 4가지 유형의 보호 격차를 식별하였다.

특히 NO_SUGGESTIONS 플래그가 다른 플래그와 함께 사용될 때 의도적으로 무시되는 동작은, 안드로이드 표준 보안 플래그가 키보드 구현체에 의해 무력화될 수 있음을 보여주는 사례이다.

본 연구는 이러한 격차를 해결하는 보안 강화 패치를 제안하며, PoC 기반 정량 평가를 통해 그 효과를 검증할 예정이다.

## 부록 A: 진행 상태

### 완료된 작업

- 빌드 환경 세팅 (macOS, Android Studio)
- 에뮬레이터에 키보드 설치 및 IME 활성화
- Git 저장소 fork 및 브랜치 분리
- 코드 진입점 파악 (onStartInput → onStartInputView)
- 상속 체인 6단계 추적
- 두 보호 메커니즘 발견 및 분석
- 6개 취약점 식별 및 코드 증거 확보
- 보호 정책 비일관성 분석

### 진행 예정 작업

- PoC 안드로이드 테스트 앱 제작 (4주차)
- 취약점 입증 실험 (4주차)
- 보안 강화 패치 구현 (5~7주차, B 담당)
- 강화 전후 비교 평가 (8주차)
- 최종 보고서 및 발표 자료 (9~10주차)
